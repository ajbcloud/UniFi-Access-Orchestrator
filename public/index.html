What this wizard does
In a few short steps we'll connect to your UniFi Access controller, confirm it can see your doors and users, and help you create your first automation rule. Every step has a short explanation, and you can leave at any point with Skip for now and come back later from the Setup button in the top bar.
Before you start, have this ready:
  • Your UniFi Access controller powered on and on the same network.
  • An API token from the Access portal (we show you where to get it).

Test Tools
Test Configured Rules

Pick a real user and door, run a preflight check, and simulate the rule end-to-end through the same path a real badge tap takes. By default no controller unlock is sent — flip Execute real unlock to verify the door physically opens.

Why a passing test can still fail at the door
  • Reader / hub offline — door is in discovery cache but the controller reports it not bound to a hub.
  • Controller auth — token lacks Locations:Edit, so unlock writes return 401/403 even though discovery works.
  • Self-trigger filter — the orchestrator's own unlock event (with the configured marker) is intentionally ignored.
  • Group not resolved — the user's UniFi group name isn't mapped in resolver.unifi_group_to_group, so no rule can match.
  • No matching rule for that trigger door — a rule for the group exists, but its trigger door doesn't match where the user actually tapped.
  • WebSocket disconnected — when event source = WebSocket, real taps never arrive if the socket isn't OPEN.

Admin API key required

This dashboard is protected by an admin API key. Find it in config.json under server.admin_api_key (File menu, Open Config File in the desktop app), or in the service startup log. It only needs to be entered once per browser.
